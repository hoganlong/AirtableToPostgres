# Schema Customization Feature Guide

## Overview

The AirtableToPostgres tool now supports two modes:

1. **JSONB Mode (Legacy)**: Stores all Airtable fields in a single JSONB column
2. **Custom Schema Mode (New)**: Creates PostgreSQL tables with typed columns for each Airtable field

## Quick Start

### 1. Enable Custom Schema Mode

Edit `appsettings.json`:

```json
{
  "Airtable": {
    "ApiKey": "YOUR_AIRTABLE_API_KEY",
    "BaseId": "YOUR_BASE_ID",
    "TableName": "ARTWORK"
  },
  "PostgreSQL": {
    "ConnectionString": "Host=localhost;Port=5432;Database=your_db;Username=user;Password=pass"
  },
  "Schema": {
    "UseCustomSchema": true,
    "SchemaFilePath": "airtable_schema.txt"
  }
}
```

### 2. Ensure Schema File Exists

The `airtable_schema.txt` file should be in the project root. It's generated by the schema export tool and contains all table and field definitions.

### 3. Run the Sync

```bash
dotnet run
```

### 4. Verify the Results

Connect to PostgreSQL and check the generated table:

```sql
-- See the table structure
SELECT column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_name = 'artwork'
ORDER BY ordinal_position;

-- View sample data
SELECT * FROM artwork LIMIT 5;

-- Check record count
SELECT COUNT(*) FROM artwork;
```

## Configuration Options

### Schema Section

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `UseCustomSchema` | boolean | `false` | Enable custom schema mode (true) or use legacy JSONB mode (false) |
| `SchemaFilePath` | string | `"airtable_schema.txt"` | Path to the schema definition file |
| `TypeMappings` | object | `{}` | Custom type mapping overrides (optional) |

### Custom Type Mappings (Advanced)

Override default type mappings:

```json
"Schema": {
  "UseCustomSchema": true,
  "TypeMappings": {
    "singleLineText": "VARCHAR(500)",
    "multilineText": "TEXT",
    "number": "DECIMAL(10,2)"
  }
}
```

## Type Mapping Reference

| Airtable Type | Default PostgreSQL Type | Notes |
|---------------|------------------------|-------|
| autoNumber | INTEGER | Auto-incrementing number |
| singleLineText | TEXT | Short text field |
| multilineText | TEXT | Long text field |
| number | NUMERIC or INTEGER | INTEGER if precision=0 |
| currency | NUMERIC(19,4) | Standard money precision |
| date | DATE | Date only (no time) |
| createdTime | TIMESTAMP WITH TIME ZONE | Record creation timestamp |
| singleSelect | TEXT | Single choice from list |
| url | TEXT | URL field |
| formula | TEXT or INTEGER | Depends on formula result type |
| count | INTEGER | Count of linked records |
| multipleRecordLinks | JSONB | Array of record IDs |
| multipleAttachments | JSONB | Array of attachment objects |
| multipleLookupValues | JSONB | Array of values from linked records |

## Column Name Sanitization

Airtable field names are sanitized for PostgreSQL compatibility:

| Airtable Name | PostgreSQL Column |
|---------------|-------------------|
| ID | id_field |
| TITLE | title |
| Code (from TYPE) | code_from_type |
| REFERENCE_IMAGE | reference_image |
| CREATE_DT | create_dt |

Rules:
- Converted to lowercase
- Spaces replaced with underscores
- Parentheses content extracted: "(from TYPE)" → "_from_type"
- Special characters replaced with underscores
- Reserved keywords prefixed with "at_"

## Table Structure

Every table includes these standard columns:

| Column | Type | Description |
|--------|------|-------------|
| `id` | SERIAL PRIMARY KEY | Internal PostgreSQL ID |
| `airtable_id` | VARCHAR(255) UNIQUE | Airtable record ID |
| *custom fields* | *varies* | Your Airtable fields |
| `created_time` | TIMESTAMP WITH TIME ZONE | From Airtable createdTime |
| `synced_at` | TIMESTAMP WITH TIME ZONE | Last sync timestamp |

An index is automatically created on `airtable_id` for fast upserts.

## Upsert Behavior

The sync uses PostgreSQL's `ON CONFLICT` for idempotent syncs:

```sql
INSERT INTO artwork (airtable_id, title, ...)
VALUES ('rec123', 'My Artwork', ...)
ON CONFLICT (airtable_id)
DO UPDATE SET
  title = EXCLUDED.title,
  ...
  synced_at = EXCLUDED.synced_at;
```

This means:
- Running the sync multiple times is safe
- Existing records are updated with latest data
- No duplicates are created
- The `synced_at` timestamp shows when each record was last synced

## Example: ARTWORK Table

The ARTWORK table with 23 fields generates this schema:

```sql
CREATE TABLE IF NOT EXISTS artwork (
    id SERIAL PRIMARY KEY,
    airtable_id VARCHAR(255) UNIQUE NOT NULL,
    id_field INTEGER NULL,
    reference_image JSONB NULL,
    ifilename TEXT NULL,
    title TEXT NULL,
    series TEXT NULL,
    create_dt DATE NULL,
    paper_inventory TEXT NULL,
    type_id JSONB NULL,
    medium TEXT NULL,
    dimensions TEXT NULL,
    folded_dimensions TEXT NULL,
    install TEXT NULL,
    location TEXT NULL,
    notes TEXT NULL,
    sold JSONB NULL,
    type_number INTEGER NULL,
    human_readable_id TEXT NULL,
    frontcount INTEGER NULL,
    backcount INTEGER NULL,
    photo_id JSONB NULL,
    artwork_image_id JSONB NULL,
    exhibition TEXT NULL,
    condition TEXT NULL,
    rec_create_dt TIMESTAMP WITH TIME ZONE NULL,
    code_from_type JSONB NULL,
    created_time TIMESTAMP WITH TIME ZONE NOT NULL,
    synced_at TIMESTAMP WITH TIME ZONE NOT NULL
);
```

Sample query:

```sql
-- Get artworks with their titles and creation dates
SELECT
    id_field,
    title,
    series,
    create_dt,
    medium,
    dimensions,
    location
FROM artwork
WHERE create_dt > '2020-01-01'
ORDER BY create_dt DESC;

-- Count artworks by location
SELECT
    location,
    COUNT(*) as count
FROM artwork
WHERE location IS NOT NULL
GROUP BY location
ORDER BY count DESC;

-- Find artworks with attachments
SELECT
    title,
    jsonb_array_length(reference_image) as image_count
FROM artwork
WHERE reference_image IS NOT NULL
  AND jsonb_array_length(reference_image) > 0;
```

## Working with JSONB Fields

Complex fields (links, attachments, lookups) are stored as JSONB. Use PostgreSQL's JSONB operators:

```sql
-- Extract data from JSONB fields
SELECT
    title,
    type_id->0 as first_type_id,
    jsonb_array_length(photo_id) as photo_count
FROM artwork
WHERE type_id IS NOT NULL;

-- Filter by JSONB content
SELECT title
FROM artwork
WHERE type_id ? 'rec123';  -- Contains specific record ID

-- Expand JSONB arrays
SELECT
    title,
    jsonb_array_elements_text(type_id) as type_record_id
FROM artwork
WHERE type_id IS NOT NULL;
```

## Switching Between Modes

### From JSONB to Custom Schema

1. Update `appsettings.json`: Set `UseCustomSchema: true`
2. Run the sync
3. New table will be created with typed columns
4. Old JSONB table (`airtable_records`) remains unchanged

### From Custom Schema to JSONB

1. Update `appsettings.json`: Set `UseCustomSchema: false`
2. Run the sync
3. Uses the `airtable_records` table with JSONB storage
4. Custom schema tables remain unchanged

Both tables can coexist for migration or comparison purposes.

## Troubleshooting

### Error: Table not found in schema file

**Problem**: The TableName in config doesn't match any table in the schema file.

**Solution**: Check the table name spelling. Run the tool to see available tables:
```bash
dotnet run
```
Look for: "Available tables: ARTWORK, PHOTO, SOLD, ..."

### Error: Schema file not found

**Problem**: The schema file path is incorrect or the file doesn't exist.

**Solution**:
- Verify `airtable_schema.txt` exists in the project root
- Check `SchemaFilePath` in appsettings.json
- Ensure the file is being copied to the output directory

### Warning: Unknown field type

**Problem**: A field type in Airtable isn't recognized.

**Solution**: The field will default to TEXT. You can add a custom mapping:
```json
"TypeMappings": {
  "unknownType": "TEXT"
}
```

### Error: Could not parse field value

**Problem**: A field value couldn't be converted to the target PostgreSQL type.

**Solution**:
- Check the console for specific field name
- The value will be stored as NULL
- Consider changing the type mapping to TEXT for more flexibility

### Records not updating

**Problem**: Changes in Airtable aren't reflected in PostgreSQL.

**Solution**:
- Verify the sync is running successfully
- Check `synced_at` timestamp: `SELECT MAX(synced_at) FROM artwork;`
- Ensure no errors in console output
- Try deleting and re-running: The upsert logic will recreate records

## Performance Tips

1. **Use appropriate types**: INTEGER and DATE are more efficient than TEXT for numeric/date queries
2. **Index JSONB fields**: For frequently queried JSONB data:
   ```sql
   CREATE INDEX idx_artwork_type_id ON artwork USING GIN (type_id);
   ```
3. **Batch queries**: When querying many records, use LIMIT and pagination
4. **Connection pooling**: For high-frequency syncs, configure Npgsql connection pooling
